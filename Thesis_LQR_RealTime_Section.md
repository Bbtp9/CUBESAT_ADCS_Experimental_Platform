# 1. Optimal Attitude Control via Linear Quadratic Regulator (LQR)

## 1.1. Introduction to the LQR Strategy
The Linear Quadratic Regulator (LQR) is an optimal multivariable feedback control technique widely used in modern aerospace applications. Unlike classic Proportional-Integral-Derivative (PID) controllers where gains are tuned heuristically, LQR computes the optimal feedback gains by minimizing a defined cost function. This approach is highly advantageous for a CubeSat attitude control system, as it provides a systematic way to balance the trade-off between aggressive attitude pointing (minimizing tracking error) and actuation energy constraints (reaction wheel torque limits). 

The LQR algorithm seeks to find the optimal control input $u(t) = -Kx(t)$ that minimizes the infinite-horizon quadratic cost functional:
$$ J = \int_{0}^{\infty} \left( x(t)^T Q x(t) + u(t)^T R u(t) \right) dt $$
where $x(t)$ is the state vector, $u(t)$ is the control input, $Q$ is a positive semi-definite state weighting matrix, and $R$ is a positive definite control weighting matrix.

## 1.2. Mathematical Formulation for the CubeSat System
For the single-axis attitude control, the rotational rigid-body dynamics governed by the reaction wheel can be approximated as a double-integrator system:
$$ J \dot{\omega} = \tau_{sat} $$
where $J$ is the CubeSat moment of inertia, $\omega$ is the angular velocity, and $\tau_{sat}$ is the control torque applied to the satellite body (equal and opposite to the torque generated by the reaction wheel motor).

### 1.2.1. Pointing Control Mode
For the pointing phase, we seek to drive both the attitude angle $\theta$ to a specific target $\theta_{ref}$ and the angular velocity $\omega$ to zero. The state vector is defined as tracking error: $x = [\theta_{error}, \omega]^T$, where $\theta_{error} = \theta - \theta_{ref}$.
The continuous-time linear state-space model is formulated as:
$$ \dot{x} = A x + B u $$
$$ \begin{bmatrix} \dot{\theta} \\ \dot{\omega} \end{bmatrix} = \begin{bmatrix} 0 & 1 \\ 0 & 0 \end{bmatrix} \begin{bmatrix} \theta \\ \omega \end{bmatrix} + \begin{bmatrix} 0 \\ \frac{1}{J} \end{bmatrix} \tau_{sat} $$

A custom MATLAB script (`init_simulation_lqr.m`) constructs these $A$ and $B$ matrices and leverages the `lqr()` solver to synthesize the $1 \times 2$ matrix gain $K_p$. Mathematically, applying optimal control to a double-integrator system fundamentally yields the structure of a Proportional-Derivative (PD) controller, with the intrinsic advantage that the $K_{prop}$ and $K_{deriv}$ components are mathematically optimized to respect the physical limits defined by the user in matrices $Q$ and $R$.

### 1.2.2. Detumbling Control Mode
During the initial deployment of the CubeSat from the launch vehicle dispenser, high angular rates may be present. The detumbling phase prioritizes rate damping over arbitrary pointing. Thus, the system is reduced to a single state $x_d = [\omega]$. 
Using equivalent matrices $A_d = 0$ and $B_d = \frac{1}{J}$, a separate scalar optimal gain $K_d$ is computed. The dual-LQR architecture allows for specialized weighting matrices: prioritizing speed reduction heavily during detumbling, and precision alignment during pointing.

## 1.3. Simulink Software Architecture
The LQR strategy was seamlessly integrated into a dynamic simulation using an automated model generation methodology (`build_lqr_model.m`). Instead of discrete arithmetic sum blocks, the closed-loop Simulink architecture (`Cubesat_Control_LQR.slx`) was constructed to reflect true state-space principles:
1. **Signal Vectorization**: The angular error $(\theta_{ref} - \theta)$ and inverted angular rate $(-\omega)$ are multiplexed into a comprehensive state-error vector.
2. **Matrix Feedback**: This state vector is multiplied by the optimized row vector $K_p$ using a Matrix Gain block computationally mimicking $u = -Kx$.
3. **Automated Hysteresis Logic**: A relay-based logic gate actively monitors incoming angular velocity. If the velocity crosses safe pointing thresholds, the architecture dynamically switches the actuator source from the Pointing LQR signal back to the Detumbling scalar signal.

# 2. Hardware-in-the-Loop Concept: Real-Time Telemetry Simulation

## 2.1. Objective of the Live Visualization
Standard computational environments instantly solve and plot Ordinary Differential Equations (ODEs). However, real-world CubeSat ground stations receive telemetry data sequentially over time via a communication downlink. To bridge the gap between simulation and physical hardware testing, a real-time visualization environment was developed (`plot_cubesat_results_rt.m`).

## 2.2. Methodological Approach
The implementation aims to simulate the live data stream exactly as it would appear on a terminal when the physical CubeSat is built and operational.
1. **Progressive Rendering (`animatedline`)**: Instead of rendering a static plot comprising all datasets simultaneously, the script initializes specialized dynamic plot objects.
2. **Time-Scaled Logical Indexing**: A constrained `while` loop runs synchronous to a localized clock (`tic/toc`). Time stamps outputted by the previously executed Simulink model are compared against the chronometer. Data points are strictly appended to the visual interface only when the corresponding simulated time logically elapses.
3. **Dynamic State Indication**: In tandem with plotting signals such as Reaction Wheel Torque and Attitude Angle, the interface autonomously highlights the real-time operational mode (e.g., actively flashing "DETUMBLING" in red, then shifting to "POINTING" in turquoise when the mode transition logic actuates).

This provides an invaluable visual testing ground for future ground station dashboard development and verifies that the dynamic transient responses of the LQR framework behave nominally over time.
